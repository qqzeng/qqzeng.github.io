hashicorp go-plugin 解析

发表于 2021-12-19 | 分类于 go
字数统计: 2,375 words | 阅读时长 ≈ 8 mins

通常情况下 Go 采用静态编译方式来构建程序，这使得构建出来的单个二进制包含程序运行的所有依赖信息，即不需要依赖操作系统运行时库，因此可以直接在绝大多数机器上运行。在 Go1.8 以前，Go 缺乏动态加载代码机制，因此不能实现类似于 Java 的插件系统。Go1.8 引入的 go plugin使得 Go 支持动态链接，go plugin 本质是将 Go 包编译为 so 文件共享库，然后主程序可以在运行时动态加载这些提前编译好的 so 文件，以从中获取并使用共享库中导出的变量或者函数，因此我们终于可以基于 go plugin 构建支持热插拔的插件系统。但 go plugin 有一些缺陷，比如官方只明确说明其支持 Linux、FreeBSD 以及 macOS，因此在其他平台上可能无法使用或者有潜在风险，这也是 go plugin 特性自发布以来一直未得到广泛使用的原因之一。Hashicorp go-plugin 则基于完全不同的原理开发，它并未将插件代码真正 load 到主程序中，而是将进程间通信作为主程序调用插件导出函数的实现方案，这使得插件库和主程序完全独立，插件崩溃也不影响主程序的稳定性。但性能方面，Go 原生的基于动态链接的方式肯定更优秀。

go plugin 的缺点

Go 的插件系统本质上是基于 C 的动态库实现，因此它肯定存在 C 动态库的缺点。从使用 Go plugin 的角度，这里非常细致地阐述了 Go 插件系统本身的局限性，我这里直接做出总结，有兴趣的读者可以去读原文。

  • 官方只明确说支持 Linux、FreeBSD 和 macOS，因此其他系统是否支持不能确定，且即使可以使用，会不会有潜藏风险无从得知；
  • 若主程序和插件包依赖同一第三方库，则该共同依赖包的版本必须一致；
  • 若主程序或插件库任一一方采用 vendor 构建，则主程序和插件包必须基于同一个 vendor 目录构建；
  • 主程序和插件库使用的编译器的版本必须一致；
  • 使用插件库的主程序只能使用动态链接，不能使用静态链接；
  • 关于插件的版本，若插件名相同，内容也相同，主程序多次 load 不会出现问题；但若插件名相同，但内容不同，主程序运行时多次 load 会导致无法恢复的 panic。因此插件版本管理需要注意。

正如原作者所说，若想使用 go plugin，则必须要保证主程序和插件库的构建环境一致，且对依赖的第三方包的版本一致。若希望在项目中大规模、系统性的使用 go plugin，可以借鉴业界 go plugin 实践的开源项目的最佳范例，如 gosh、tidb 以及 tyk api gateway，其中 tidb 还给了完整的设计方案。

Hashicorp go-plugin

Hashicorp go-plugin 概述

go-pluign 是 Hashicorp 公司主导的开源项目，2013 年正在开发的 Packer 项目需要使用插件，但当时 Go 还未推出动态加载插件库特性，因此 Hashicorp 公司就自己开始着手实现。另外，在其开源的 Nomad、Consul、Vault 和 Terrform 等项目中也都有应用。

Hashicorp go-pluign 和 Go 原生插件库的实现方式完全不同。go-plugin 实现插件库的基本原理是多进程间的 RPC 通信，即将主程序调用插件库方法实现为同一台主机上的主程序进程和插件进程之间的通信，网络协议可以是 Go 标准库的 net/rpc 或 gRPC，一般推荐使用 gRPC。而底层使用的协议类型可以是 TCP 或者 Unix 域套接字，引入 Unix 域套接字的原因是在实际使用时，机器上有一些端口被防火墙限制，因此基于 TCP 的方式无法工作。go-plugin 插件一般由宿主进程（主程序）调用，宿主进程以插件二进制文件为映像创建子进程，然后通过单个网络连接与之通信，之所以强调通信只需建立单个网络连接，是因为后期随着 go-plugin 的迭代，其在 v3 版本支持连接的多路复用。显然，我们可以发现 go-plugin 的设计原理使得它没有 Go 原生插件系统的那些缺陷。

go-plugin 设计之初的目标是实现良好的扩展性、易用性、方便安装和管理，同时插件进程不会影响到主程序的稳定性。在其官方的 README 中总结了其核心特性，其中有一些关键项：

  • 编写插件即实现 Go 接口。对于插件的开发者而言，他只需要实现特定接口，而针对插件调用方而言，可像调用本地方法一样调用插件预先声明的接口，go-plugin 会在中间处理本地调用转换为 RPC 调用的细节；
  • 跨语言支持。插件可以基于任何主流语言编写和消费，因为插件和主程序是基于 RPC 通信；
  • 支持复杂参数。支持将 interface、io.Reader/Writer 等类型作为被调用接口的参数；
  • 宿主进程独立升级。宿主进程可独立升级，而不影响插件进程，且在升级完毕后，插件进程可关联到新的宿主进程；
  • 宿主进程稳定性。即插件进程奔溃不会影响宿主机进程；
  • 同步 stdout 和 stderr：插件实际是以主程序的子进程在运行，这些子进程所使用标准输出/错误会被自动同步到宿主进程，同时，宿主进程可以为同步的日志指定一个 io.Writer。

Hashicorp go-plugin 基本原理

宿主机进程，也即主程序访问插件库的过程本质为一个 RPC 通信过程，而 go-plugin 的核心能力即类似一个 RPC 库。但显然其还需处理些额外的事情，比如连接复用、stdout 和 stderr 的在两个进程间的同步等。

插件接口是宿主进程和插件进程进行通信的桥梁，因此它需要被放到一个共享的仓库。从 go-plugin 库底层实现来看，在宿主机和插件进程正式建立连接通信前，他们之间的交互逻辑如下：

  • 插件进程会将同宿主机进程握手的信息输出到 stdout，具体包含 go-plugin 协议版本、应用协议版本、网络类型、网络地址以及协议类型；
  • 然后就开启监听，等待该地址上的客户端连接到来；
  • 宿主机进程则首先启动插件二进制程序；
  • 然后读取和校验插件进程输出的 stdout 中的握手信息；
  • 然后基于握手信息包含的套接字构建 gRPC Client，并同插件进程的 gRPC Server 建立连接；
  • 最后将 gRPC Client 转换为插件实例，用于后续的调用。

另外，结合官方的 go-plugin 示例，从使用的角度来看，宿主机进程和插件进程通信过程大概如下：

  • 插件进程声明自己支持的插件集，插件集中包含每个插件以及其对应的业务实现。插件进程类似于服务提供方，即 Server 端；
  • 插件进程会调用 Serve 方法，创建服务器端，并将插件声明的接口的实现委托给底层的业务接口实现；
  • 宿主机进程则类似于 Client 端，其只需将插件接口实现放到自己插件集中，以声明自己需要消费调用的插件；
  • 其首先通过获取插件客户端以启动插件进程并进一步得到插件进程监听的套接字、构造 gRPC client；
  • 然后宿主机进程将 gRPC client 转换为插件实例；
  • 当宿主机进程调用插件实例的方法时，go-pluign 会将宿主机进程对插件的调用转换为 gRPC 请求，通过 TCP 或则 Unix 域套接字将请求送到 Server 端（插件进程关联的 gRPC Server），请求被 gRPC Server 解析并将其中包含的实际方法调用转给业务接口的实现来完成，最后将业务接口的返回体封装作为对 gRPC Client 的响应，gRPC Client 获取响应后同样会解析响应内容并返回给宿主机进程。
