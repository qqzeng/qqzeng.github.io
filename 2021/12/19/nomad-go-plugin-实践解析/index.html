nomad go-plugin 实践解析

发表于 2021-12-19 | 分类于 go
字数统计: 2,462 words | 阅读时长 ≈ 9 mins

Nomad 是 Hashicorp 公司出品的一款用于编排和管理应用的框架的开源产品。所谓编排即用于组织应用及其依赖的服务，并实现应用的多实例（副本）机制，而所谓管理即管理应用的生命周期，提供类似于操作系统 supervisor 进程之于其他进程的功能。Nomad 支持多种类型的负载，如批处理任务、容器化应用又或者 Java 应用等。在设计上，Nomad 通过任务执行驱动特性抽象了负载类型，即每一种负载类型由同其相对应的任务执行驱动来负责运行，这使得用户可自定义任务驱动以提供符合其特殊需求应用的运行时环境。实际上，可扩展的任务执行驱动即为 Nomad 的插件系统，每一种类型的执行驱动即为一个插件，其底层基于 go-plugin 实现。简单而言，Nomad 在启动时会同时运行其内置的插件二进制集合，当用户提交特定类型的负载到集群时，其通过 gRPC 同相应类型的插件程序通信，以执行用户的任务。承接上一篇查询 go-plugin 的文章，为了更好地理解 go-plugin 的实际应用，本文简单介绍 go-plugin 在 Nomad 中的实践。

Nomad 任务执行驱动

Nomad 基于 go-plugin 将每一种负载类型的执行环境抽象为任务执行驱动。所谓任务执行驱动，即为基于 go-plugin 开发的插件系统中的业务实现，Nomad 官方定义了所有扩展的执行驱动所必须实现的接口，这些接口实际上即为一个任务的生命周期操作接口。具体地，BasePlugin 定义驱动的基本信息和配置，因此是需要实现的接口的一部分，而 DriverPlugin 即为业务接口，其继承了 BasePlugin，同时定义了任务的基本操作接口，如启动任务、停止任务、恢复任务等。还记得上一篇文章阐述的，基于 go-plugin 开插件系统，对于服务提供方（Server 端），只需提供业务实现即可，其他的如请求转发和解析逻辑都由 go-plugin 完成。因此，若我们需要让 Nomad 支持我们自定义的负载类型，只需要编写 DriverPlugin 的实现即可，如下图中的 SvcRun 即为一个可运行常驻型应用的任务驱动，另外几个都是 Nomad 官方内置的执行驱动。关于 Nomad 任务执行驱动可以从这里详细了解。

Nomad 任务执行驱动初始化解析

本文不会过度关注任务执行驱动各接口的定义和实现，而是专注于解析 Nomad 如何基于 go-plugin 来实现任务执行驱动系统，实际上是一个官方的 go-plugin 的实践案例，其本质和 go-plugin 的 helloworld 案例相同。

Nomad 的任务驱动包含两个部分：内部驱动和外部驱动，内部驱动即官方内置在 Nomad 主程序中的驱动，因此其实际上内嵌在 Nomad 主程序二进制中的。而外部驱动为用户自定义的驱动，二者在 Nomad 中的初始化逻辑不同，因为外部驱动 Nomad 主程序无法提前知晓，而只能在启动时，扫描驱动目录以加载并执行用户自定义的驱动二进制。总而言之，用户自定义驱动才是 Nomad 基于 go-plugin 的实践。

外部驱动 Server 端初始化过程

首先来阐述用户自定义驱动 Server 端的初始化过程，即插件进程自身的初始化过程。用户自定义的驱动包含两部分，一个是针对 DriverPlugin 的实现，一个是 main 方法用于启动插件二进制，并开启 gRPC Server 监听等待宿主机进程的连接。在代码层面，其过程比较简单：

  1. 在 main 方法中调用 Nomad 的 Serve 方法同时传入任务驱动的构造工厂方法；
  2. 在 Serve 中实例化该任务驱动；
  3. 最后根据插件类型（Nomad 基于 go-plugin 实现了任务驱动插件 DriverPlugin 和设备驱动插件 DevicePlugin）调用 go-plugin 的 Serve 方法。其传入了握手信息、其支持的插件（BasePlugin 和 DriverPlugin）并将网络协议设置为 gRPC；
  4. 后续逻辑即为 go-plugin Server 端初始化逻辑：启动插件二进制，同时构造 gRPC Server 并开启连接监听。

任务执行驱动加载器的初始化过程

考虑到内部驱动的代码是放在 Nomad 主程序中的，因此其没有外部驱动被动加载执行这个过程，其在主程序包初始化时会通过插件的 Catalog 的接口 主动注册到宿主进程的插件集中，注册时同样提供其驱动配置以及构造工厂方法。在 Nomad 主程序启动时会遍历所有主动注册的插件，对他们进行初始化。下面简单阐述 Nomad 插件加载器的初始化过程。

  1. 在 Nomad 主程序启动时，其分别初始化 Nomad Client 和 Server 以及 Consul Client 的配置，并引导 Client 的插件加载器系统。引导插件加载器主要包含如下几个步骤：
  2. 首先通过 Catalog 获取所有主动注册的内部驱动；
  3. 然后初始化插件加载器的配置 PluginLoaderConfig，其中包含内部驱动的配置以及外部驱动的目录等信息，依次来实例化插件加载器 PluginLoader；
  4. 在实例化 PluginLoader 的过程中，首先校验插件加载器包含的各项配置，并转换插件系统中支持的驱动版本，然后构建插件加载器，并调用它的 init 方法来初始化插件加载器；初始化过程主要包含几个步骤：
  5. 首先初始化内部驱动，即依次构建内部驱动实例，通过调用驱动实例获取驱动基本信息，然后获取驱动的版本以及配置 schema，最后将这些信息封装成 pluginInfo 返回；
  6. 其次，扫描外部驱动的二进制目录，获取合法的驱动二进制信息；
  7. 接下来，基于 go-plugin 加载所有外部插件，以获取其基础信息。该过程即为 go-plugin 宿主机进程的初始化过程，核心步骤是创建插件客户端，以启动插件二进制程序，并获取 gRPC Client，然后将其转换为基础驱动实例以获取驱动的基本信息和配置信息。最后同样将这些封装为 pluginInfo 返回；
  8. 然后，合并内部和外部驱动实例，若包含相同的驱动，取版本较高的驱动实例；

  9. 最后，校验并设置驱动的配置信息，即针对每一个驱动，若其包含用户定义的配置信息，则基于其配置 schema 解析它，并调用驱动的 SetConfig 方法以设置驱动的配置，这些配置信息在驱动执行

  10. 插件加载器 PluginLoader 实例化完成后，然后基于此 PluginLoader 构建 SingletonLoader，SingletonLoader 用于实现保证每一个驱动只存在一个实例，其内部保存了一个以驱动实例的 ID 作为 Key 的字典。宿主机进程可以通过传入驱动的基础信息和配置信息以实例化一个驱动（Dispense 接口），后续则直接使用已实例化的驱动即可。在宿主机进程当即重启时，其也可通过宕机为该驱动实例保存的基本信息（网络协议、通信地址以及驱动进程 ID）来重新让驱动进程关联到新的宿主机进程（Reattach 接口）。实际上，SingletonLoader 即为 Nomad 内部封装的用于获取驱动实例的入口。

Nomad 执行任务时获取驱动过程解析

当用户提交一个具有特定任务驱动的任务时，该任务实例在 Nomad 中是以 Allocation 承载的，Nomad Client 会定期从 Nomad Server 同步 Allocation 实例，根据对比 Nomad Client 已有实例，针对性处理拉取到的 Allocation，比如当获取到新的 Allocation 实例时，其会创建新的 AllocationRunner，并交给 AllocRunner 去处理，而 AllocRunner 会取出 Allocation 中包含的所有任务配置，根据任务的配置信息，生成对应的任务，然后创建对应的 TaskRunner 实例，并将任务交给 TaskRunner 处理。TaskRunner 初始化时，会初始化其需执行的任务的驱动实例。具体地，其将任务包含的驱动的名称传给 DriverManager 以获取一个指定类型的驱动实例，DriverManager 底层则是通过上文阐述的 SingletonLoader 来获取（Dispense 或者Reattach）对应的驱动实例。

至此，Nomad 基于 go-plugin 的实践相关内容以及阐述完毕。本文主要是介绍了三个方面的内容，Nomad 任务执行驱动的简要介绍、Nomad 外部驱动 Server 端初始化过程、Nomad 启动时任务驱动加载器的初始化过程以及 Nomad 执行任务时获取驱动实例的过程。实际上，Nomad 的任务执行驱动系统同 go-plugin 相关的逻辑和 go-plugin 的 helloworld 项目的逻辑完全一样，只不过因为 Nomad 项目本身的复杂性，宿主机进程获取并管理外部插件实例的逻辑较为复杂，但这同 go-plugin 本身的使用并无太大关联。
