memberlist 原理剖析

发表于 2020-09-19 | 分类于 分布式系统 ， 组成员协议
字数统计: 7,536 words | 阅读时长 ≈ 26 mins

在此之前，我们介绍了组成员资格协议 WSIM，其主要包含失败检测和基于 Gossip 的传播更新两个部分。随后阐述了 Hashicorp 研究人员在 WSIM 的基础之上引入 local health 概念而提出的 Lifeguard，以降低节点负载较高的情况出现的高误判率，即将健康成员声明为故障成员，Lifeguard 从三个方面对 WSIM 进行针对性改善和拓展。前两篇文章侧重于以论文内容为基础，并从协议设计的（理论）角度进行阐述，以了解协议设计思想以及基本运行流程。而本文则深入到源码，并从协议架构以及实现优化的（实践）角度来分析，以了解其关键实现细节以及工程落地的权衡优化。memberlist 是 Hashicorp 在以 Lifeguard 增强 WSIM 后开源实现的组成员资格协议库，被用于 Consul 和 Nomad 开源项目。

本文并非详细的源码解读教程，只阐述关键数据结构和协议运行流程。因此，若读者有兴趣，可在参阅本文之后详细阅读源码。另外，建议读者在阅读本文前，先了解协议的基本理论和设计规范，这有助于读者理解源码，可参考这里和这里。最后，本文的分析基于版本 commit id 为 9e07daff43。但源码注释基于版本 v0.2.4。

本文侧重于从协议实现角度来阐述。具体地，本文首先阐述协议框架的总体执行流程，这有助于把握自行阅读源码时的方向和切入点。其次，理清协议包含的消息的关键处理逻辑。然后，针对性指出 Lifeguard 优化如何落地。最后，简单阐述如何建立协议库 memberlist 和上层应用系统（如 Serf）之间的关联。

协议库模块划分

memberlist 协议库源码结构比较清晰，代码量也不大，协议本身也基本无第三方库依赖。关键组件或模块归纳如下：

  • memberlist.go 为协议库的入口文件，定义了 Memberlist 核心结构，其包含集群中的成员需要维护的运行时结构状态，更侧重于辅助协议实现的字段结构。相应地，其也包含了几个供上层应用调用接入协议层的接口，如启动协议库、关停协议库，以及获取集群成员。另外，也包含导致集群成员变更的操作，如新节点加入集群、成员离开集群。但其并非唯一的同上层应用交互的入口文件，具体在第四小节阐述；

  • config.go 为协议库的核心配置文件，定义 Config 核心结构，不同于 Memberlist，其侧重于维护协议相关的状态或配置，以及上层应用 hook 到协议库的入口，另外包含一些网络配置相关的 help 方法；

  • net.go 为协议库消息中间层处理的文件，定义了大部分消息的结构，如 ping、indirectPing、alive、dead 以及 suspect 消息等，相应地，其也包含对应消息的 handler，其消息处理的逻辑侧重于消息的编解码、消息的合成和分解、消息的压缩和解压缩以及消息处理的 handler 分发，总之，其类似于消息处理的一个中间层，处理逻辑较少或不涉及消息或协议相关的状态；
  • state.go 同样为协议库消息处理的核心文件，其主要包含各类消息的 handler 的核心逻辑，即涉及到协议相关状态变更的处理逻辑，如 ping、indirectPing、alive、dead 、suspect 以及 refute 消息的 handler。另外，也包括故障检测 probe 操作、周期性成员全量状态同步 push/pull 的操作以及周期性的传播集群状态更新的 gossip 操作；
  • transport.go 和 net_transport.go 为协议库的消息传输组件，定义了协议消息传输的核心模型 Transport 接口，并提供了默认的实现 NetTransport 结构。其主要包含协议网络消息处理和传输的操作，比如针对网络连接（或网络包）的创建、监听、读取、发送以及销毁等逻辑；
  • broadcast.go 和 queue.go 两个文件用于处理集群消息广播操作，其核心是基于 b 树结构对集群需要广播消息的存储、检索以及移除操作，其中对 b 树存储消息的操作的一个核心依据是消息被转发的次数；
  • suspiction.go 用于管理 suspect 消息，比如定义成员等待可疑成员响应的超时时限的计算方法以及可疑成员被标记为失败需要得到多个成员的确认的操作逻辑等；
  • awareness.go 用于管理由 Lifeguard 引入的成员 local health 指标，以动态调整成员标记其它成员为失败的操作的可信度；
  • *_delegate.go 包含允许上层应用针对协议状态变化而针对性制定的逻辑的 hook 文件，比如，当集群中出现冲突的成员时，或者当集群成员之间的状态发生合并操作时，上层应用的针对性执行的逻辑。详细内容在第四小节阐述。

协议库基本执行框架

memberlist 组成员协议库基本执行框架同协议运行流程类似。上层应用程序（如 Serf）通过提供协议配置并调用 memberlist.go 提供的方法来创建并启动协议库（Create），这首先初始化 Memberlist 结构，然后创建网络监听和对应类型的消息的 handler。具体地，首先创建 tcp 连接处理监听并启动对应的 tcp 消息的处理循环，在 tcp 消息的 handler 中，依次从连接中读取消息，然后根据消息类型（userMsg、pushPullMsg 及 pingMsg）针对性执行对应的消息处理逻辑。其次，再创建 udp 网络包处理监听并启动对应的 udp 消息处理循环，在 udp 消息 handler 中，同样根据消息类型（compundMsg、compressMsg、pingMsg、indirectMsg、ackRespMsg、nackRespMsg、suspectMsg、aliveMsg、deadMsg 以及userMsg）进行针对性处理。

接下来，当前节点将其自身标记为 alive 状态，并向集群成员广播此 alive 消息。

最后，执行周期性的消息发送和接收操作。具体地，首先，执行一个周期性的故障检测操作，即从本地成员列表中随机选择一个成员并根据成员的状态发送探测消息，若认为对方处于 alive 状态，则发送一个 ping 消息，否则，发送一个 ping 消息和 suspect 消息，并根据响应结果进行针对性处理。其次，执行一个周期性的 pull -> push -> merge 消息操作，即随机选择一个集群成员，进行全量的节点状态的同步操作。最后，周期性的给集群中随机若干个成员并向它们发送广播消息（从排队缓存的需要被广播到集群中的消息集合中按顺序取出）。

协议消息处理

memberlist 组成员协议库包含的消息种类较多，包括 tcp 和 udp 类型的消息。有些消息的处理逻辑比较直观，有些消息处理逻辑较为复杂，下面先简要介绍集群成员在收到不同消息的后的处理逻辑，然后，介绍集群成员周期性执行的三个操作。值得注意的是，消息允许被压缩和加密。

tcp 消息处理逻辑

首先阐述通过 tcp 来交互的三种消息处理逻辑。需要注意的是，这些消息都是直接通过连接响应对端，并不会以 gossip 的方式广播到集群，也不会执行 piggyback 操作。

userMsg

当集群成员通过 tcp 连接收到 userMsg 时，其处理逻辑很简单，在解码消息后，直接回调上层应用的 hook 方法。显然，这是一种单向消息，即 userMsg 类型的消息的处理过程并不属于协议范围，相反，它是属于上层应用程序借助协议库实现的成员之间的单向传输消息。

pushPullMsg

当集群成员通过 tcp 连接收到 pushPullMsg 时，其会执行一个标准的 pull -> push -> merge 消息以同集群中一个随机选中的成员进行节点状态全量交换同步。具体地，其首先从连接中读取远程节点状态，所谓节点状态，即该节点本地的集群成员视图。然后，将本地状态发送到对端，同时也可发送上层应用程序委托的附属数据，最后，执行远程节点状态和本地节点状态的合并操作，在合并过程中，一一同步处理远程节点视图中包含的集群成员的状态，即对于 alive、left、dead 以及 suspect 状态的节点，节点会当作自身收到对应类型的消息，并执行对应消息类型的 handler，具体在后面阐述。类似地，在合并操作的同时，同样会调用上层应用的针对集群成员状态出现 merge 操作所对应的 hook 接口。

pingMsg

当集群成员通过 tcp 连接收到 ping 时，其直接返回一个 ack 消息。

udp 消息处理逻辑

其次，阐述通过 udp 来交互的十种消息的处理逻辑。

compoundMsg

compoundMsg 是一种复合消息类型，所谓复合类型消息，即其容纳有其它消息或消息集合。引入 compoundMsg 的目的是充分利用消息传输带宽，同时加速集群成员状态的收敛。当集群成员通过 tcp 连接收到 pushPullMsg 时，它首先解析出其中包含的单个基础消息，然后，再执行消息处理 handler 的逻辑。

compressMsg

compressMsg 同样是一种复合消息类型，其作用同 compoundMsg 类似。且处理过程也类似，即首先解压缩出其中包含基础消息，然后回到消息处理 handler 的逻辑。

pingMsg

当集群成员通过 udp 连接收到 pingMsg 时，其直接返回一个 ack 消息。但需要注意的是，此 ack 消息不会直接通过 udp 发送给对方，这同通过 tcp 连接收到的 pingMsg 不同，它会执行一个额外操作：构建一个 compoundMsg，并从排队缓存的广播消息集合中取出若干个消息，以尽可能使得此 compoundMsg 接近 udp 消息的额外网络包大小，最后才将消息发送给对端。

indirectPingMsg

当集群成员需要依靠其它集群成员对目标成员的状态的进行判定时，其会向若干个随机的集群成员发送 indirectPingMsg，根据它们的检测结果来间接判断目标成员的状态。具体地，当集群成员通过 udp 连接收到 indirectPingMsg 时，它会向消息中指定的目标成员发送一个 pingMsg，当目标成员在规定的超时时限内返回了 ackMsg，则其同样发送一个 ackMsg 给 indirectPingMsg 消息的发送者，否则，发送一个 nackMsg。同样，此响应的 ackMsg 或 nackMsg 也会执行 piggyback 操作。

ackRespMsg

当集群成员通过 udp 连接收到 ackRespMsg 时，它会执行之前为此消息设置的对应的 ack handler，然后将此 handler 移除。此 ack handler 若超时未被调用，也会被自动移除。

nackRespMsg

类似的，当集群成员通过 udp 连接收到 nackRespMsg 时，它会执行此消息对应的 nack handler，然后将此 handler 移除。

userMsg

当集群成员通过 udp 连接收到 userMsg 时，其处理逻辑同通过 tcp 连接收到 userMsg 的处理逻辑类似。

aliveMsg

aliveMsg 被调用时机包含四种情况：当一个节点通过协议库加入集群时，会进入处理 aliveMsg 消息的流程，即类似于节点收到一条 aliveMsg；其次，上层应用可通过节点更新接口更新节点元信息，此时同样会进入处理 aliveMsg 消息的流程；再者，当集群成员进行全量状态交换同步时，针对同步节点保存的 Alive 状态节点，节点会当作自身收到了 alilveMsg。最后一种情形是当节点从网络连接中收到一条 aliveMsg。

当集群成员收到 aliveMsg 时，它首先回调上层应用的 Alive hook。然后，判断 aliveMsg 中包含的节点是否已经保存于本地，若不存在，则将其插入到自身集群成员视图列表中的一个随机索引，否则，当节点已经存在于本地集群视图列表中时，则进一步判断 aliveMsg 中节点元信息（如 ip 地址）是否和本地保存的对应节点的元信息冲突，若产生冲突，则需进一步判断节点是否允许更新其元信息，显然，只有节点处于 left 或者 dead 状态，且协议被配置为允许节点回收（即节点的名称重复使用必须间隔指定时间），节点才会更新 aliveMsg 中包含的节点在本地存储的元信息和状态，相反若不允许更新节点状态，则回调上层应用的 Conflict hook，然后直接中止后续处理流程。接下来，节点清除对应节点的 suspicion 定时器（因为收到了 aliveMsg，明确节点处于存活状态）。同时，若发现此 aliveMsg 正是针对节点自身，且并不是节点自身在启动时加入集群时发出的，则节点可能需要执行 refute 操作，即向集群中广播 aliveMsg，以公告自己仍然处于存活状态。相反，若发现此 aliveMsg 同自身无关，或者即使此消息同自身相关，但也并非在节点启动加入集群时发出的，此时直接将此 aliveMsg 广播到集群。这两种情况的最终处理结果虽然都是向集群中广播 aliveMsg，其区别在于：在前一种情况下，节点可能会更新自身的 incarnation 计数器，同时，考虑到节点驳斥了来自其它成员的怀疑，因此需要显式地增加节点自身的 local health 值。最后，执行上层应用的相关事件的 hook 接口，比如节点的元信息更新了，或者节点状态更新了。

值得注意的是，源码中有几个关于节点的 incarnation 计数器的 Fast-path 需要考虑清楚。

suspectMsg

suspectMsg 被调用时机包含三种情况：当节点执行周期性的探测过程中，发现节点没有响应其 ping 消息和 indirectPing 消息，则会进入到处理怀疑目标节点的流程，即类似于节点收到一条 suspectMsg；其次，当节点周期性地同集群一个随机成员执行一个全量状态交换同步时，发现从同步节点那里得知某些节点处于 suspect 状态，则同样进入处理怀疑目标节点的流程；最后一种情形是当节点收到一条 suspetMsg。

当集群成员收到 suspectMsg 时，首先判断该节点是否存在于其本地集群成员视图中、其 incarnation 计数器是否过期，以及该节点的状态是否已经不是 Alive了（可能目标节点已经被当前节点标记为 Suspect 了），若是，则直接退出后续处理流程。接下来，判断当前节点是否已经为被目标节点设置对应的 suspicion 定时器。

suspicion 定时器引入的是为了更可靠地标记一个节点处于 Dead 状态，因此其必须由多个节点进行确认目标节点可能处于故障状态，而确认途径为目标节点必须在超时时限内，通过返回一个 aliveMsg来驳斥怀疑，另外，此超时时限也是动态变化的，一个直观的解释是，随着集群中越来越多的成员确认了目标成员可能处于故障状态，则此超时时限会越来越短，因为有理由认为目标节点真的处于故障状态。

若已为目标节点设置对应的 suspicion 定时器，则执行一个 confirm 操作，在 confirm 操作中，每收到一次来自集群中其它成员对目标成员的怀疑确认，则根据预定策略动态更新计时器的超时时限，而且对于集群中每一成员，我们只接收一次确认（考虑到 suspect 消息可能会被单个成员多次广播），然后将广播此 suspect 消息，若一段时间后，我们得到了集群中足够多的成员的确认，则我们不再接收任何确认。另一方面，若定时器超时，则触发对应的超时动作，即认为节点已经故障，随即进入节点故障处理流程，类似于收到一个 deadMsg 时的处理流程。

相反，若未设置对应的 suspicion 定时器，则同样会广播此 suspect 消息，然后，更新目标节点的状态为 Suspect，为其设置并启动对一个的 suspicion 定时器，同时，需要设置对应定时器超时后的动作。值得注意的是，节点有可能会发现自身就是被怀疑的对象，即此 suspect 消息时针对自身的，但显然节点仍处于存活状态，因此，会进入驳斥怀疑的流程，即向集群广播 aliveMsg 消息，同时更新自身的 local health 值。

deadMsg

deadMsg 被调用时机包含四种情况：当一个节点显式调用 Leave 接口离开集群时，会进入处理 deadMsg 消息的流程，即类似于节点收到一条 deadMsg；其次，当节点在处理 suspect 消息时，若其处理结果为确认节点真的处于故障状态了，则定时器超时触发进入处理 deadMsg 的流程；再者，当集群成员进行全量状态交换同步时，针对同步节点保存的 Dead 状态节点，节点会当作自身收到了 deadMsg。最后一种情形是当节点收到一条 deadMsg。

当集群成员收到 deadMsg 时，同收到 suspectMsg 类似，首先需要判断该节点是否存在于其本地集群成员视图中、其 incarnation 计数器是否过期，以及该节点的状态是否已经为 Dead 状态了。若是，则直接退出后续处理流程。

当然，节点会判断此 deadMsg 的目标成员是否即为自身，若发现当前的 deadMsg 确实针对的是节点自身，且节点自身仍处于存活状态（未宕机），因此，会进入驳斥怀疑的流程，即向集群广播 aliveMsg 消息，同时更新自身的 local health 值，然后返回。

另一种情况是，节点主动调用 Leave 离开集群，则此时可进一步广播此 deadMsg，以让其他成员尽可能快的知悉这一消息，使得集群状态收敛。同时设置一个接收 channel，一旦任意一个成员收到此 deadMsg 时，此节点就可以放心离开集群，否则应该在 Leave 操作中阻塞等待，直到集群成员知悉其已离开集群。然后，将节点状态标记为 Left（正常离开）。

否则，否则，若 dead 消息的节点即为发送该 dead 消息的节点，则说明该节点为主动离开集群。因此，需要将节点的状态标记为 Left 而并非 Dead，反之亦然。另外一个动作为更新本节点为目标节点保存的视图，比如节点状态的更新时间。最后回调上层应用针对节点离开集群的事件设置的 hook。

周期性触发操作

节点周期性触发执行的操作包括三个方面，其一，随机从本地集群成员视图列表中选择一个集群成员，执行故障检测的过程。 其二，随机选择一个集群成员，执行全量状态同步交换的过程。其三，随机选择若干个集群成员，执行基于 gossip 传播方式的消息传播过程。

故障检测

首先向此成员发送一个 ping 消息，同时设置 ack handler、nack handler 以及超时的 handler。

需要注意的是，此探测超时时间是动态设置的，同节点的 local health 成正相关，一个直观的解释是，节点的 local health 值越高，其越可能处于高负载状态，因此，为了顺利接收到其他成员反馈给他的消息，他需要给与目标成员更多的响应时间。

然后，根据随机选择的节点的状态发送对应类型的探测消息。若节点处于 Alive 状态，则向其发送一个 pingMsg，且此基于 udp 的 pingMsg 会通过 piggyback 操作发送出去，若节点不处于 Alive 状态，则以 pingMsg 和 suspectMsg 构建一个 compoundMsg，然后直接通过 udp 发送出去。然后等待目标节点响应或者定时器超时，若目标探测节点成功返回 ack，则在回调上层应用的 Complete hook 后，直接退出后续处理流程。否则若节点响应 nack 或定时器超时后，则继续后续的间接探测过程。

在间接探测流程中，即随机选择若干个集群成员，向他们发送基于 udp 的 indirectPingMsg。此外，在通过 udp 探测目标节点失败后，节点还可能转而使用基于 tcp 的 pingMsg，只要没有配置禁止使用 tcp 探测，则转向使用 tcp 向目标节点发送 pingMsg，同时等待响应或者超时，若基于 tcp 的 pingMsg 成功得到了响应，则返回，退出后续流程，因为这说明目标节点是存活的，但是因为其他原因（比如目标节点的网路配置，udp 端口未打开等）导致通过 udp 探测失败。否则，若通过 tcp 也探测失败，则说明目标节点可能发生故障，因此，首先更新节点自身的 local health 值，然后进入到怀疑节点（suspectNode）的操作流程。

全量状态交换同步

所谓的节点全量状态交换同步，即执行一个 push-pull-merge 操作，以交换节点同集群其他成员的集群节点成员视图状态。此操作的目的是加速集群状态收敛，不必使用基于 gossip 的消息传播方式，这在新节点加入集群，或者整个集群出现网络分区恢复的场景尤为有效。此操作的一个代价是网络带宽，因此，显然此操作不能过于频繁，特别是在集群规模较大的情况。

有两个需要注意的点是，其一，全量状态交换同步操作所包含的两个操作是基于 tcp 连接；其二，pushPull 操作的执行周期依据指定策略计算出来的。一种直观的解释是，当集群成员数目增多时，我们需要扩大同步周期，以避免整个集群网络中充斥着大量的消息。

其核心逻辑如下：节点首先随机选择一个节点，执行一个的 push-pull-merge 的操作，且如上所述此 push 和 pull 操作都基于 tcp 连接。具体而言，在 push 操作中，节点将自身本地的集群成员视图发送给对应节点，在 pull 操作中，节点从远程节点读取节点的集群视图状态，最后执行合并操作，即遍历 pull 过来的每一个节点，针对节点的状态（Alive、Suspect、Dead 以及 Left），执行对应的操作（aliveNode、suspectNode、suspectNode 以及 deadNode），需要注意的是，即使节点的状态为 dead，其仍然选择通过发送 suspectMsg 以给与节点驳斥怀疑的机会，而不是直接将节点标记为 Dead 并广播 deadMsg。 最后，回调上层应用的 Merge hook。

基于 gossip 的消息传播

所谓的基于 gossip 的消息传播，即使用一种 infection-style 的方式来传播消息，这不同于一般的中心式地消息通信架构，它可以保证在有限轮次内使得集群中的健康成员都收到这一消息。

具体逻辑比较简单，随机选择若干个节点，并从广播消息队列中取出若干消息，以构成 compoundMsg，最后依次向他们发送此 compoundMsg。

关于此 gossip 操作的目的，如论文中提到的，Lifeguard 将故障检测操作同 gossip 操作区分开来，尽管 memberlist 通过 piggyback 操作，将 gossip 消息（suspect、alive 以及 confirm）附加到故障检测消息中（ping、indirectPing 以及 ack), 但其仍然额外地周期性的发送单独的 gossip 消息，这使得可以灵活调控 gossip 消息和故障检测消息的发送频率，大多情况下，可以缩短 gossip 消息的发送周期，以加速集群状态收敛。

Lifeguard 的优化

具体的，Lifeguard 从三个方面来缓解此问题。

第一个是基于 local health 感知的故障检测，它将原本 SWIM 所使用的固定探测周期和超时时限调整为基于节点的 local health 状况来动态调整。这在 memberlist 实现中对应的实现部分为 awareness.go 模块，其包含一个动态更新的 local health（score）值，而节点的周期性探测操作的探测超时时限即通过此 score 值计算。具体地，节点的 awareness 值越大，则设置的节点探测的超时时限也越长。

第二个是基于 local health 感知的 Suspicion 方案，即动态调整 suspectMsg 的响应超时时限。这在 memberlist 实现中对应的实现部分为 suspiction.go 模块，其包含一个用于计算 suspectMsg 超时时限的策略，以及一个用于 suspectMsg 确认操作，随着节点不断收到来自集群其他成员对此 suspectMsg 的确认，其动态调整超时时限，并执行相应操作。

最后一个是扩展是 Buddy System，即节点通过 ping 消息直接将 Suspect 消息发送给目标节点，而不是通过 gossip 的方式通知目标节点，这有利于目标节点快速响应此怀疑消息。这在 memberlist 实现中对应的实现部分为周期性故障探测操作针对非 Alive 状态的节点直接发送一个 pingMsg 和 suspectMsg 构成的 compoundMsg，而不执行接收到一条 suspectMsg 的逻辑。

关联协议库和上层应用

memberlist 作为 WSIM 辅助以 Lifeguard 拓展优化后开源实现的组成员协议库，其可为上层应用提供提供诸如成员发现、失败检测以及负载均衡等功能。典型的 Hashicorp 的 Serf 即使基于 memberlist 协议库构建的。那上层应用是如何利用 memberlist 协议库的呢？或者如何建立 memberlist 协议库和上层应用直接的关联，甚至让上层应用不仅仅拥有这几个基础功能，而是能够基于事件回调的模式 hook 到协议库内部。的确，memberlist 协议库的实际充分考虑到了这些点，以最大程度的赋能上层应用。

首先，上层应用可通过实现 Delegate 接口以 hook 到协议库。Delegate 包含一些回调接口，如通过实现 NodeMeta 接口，上层应用可将节点的元信息在节点启动或更新通过发送 AliveMsg 传播出去。另外，通过实现 NotifyMsg 接口以在节点从网络连接中接收到 userMsg 时来执行特定的回调逻辑。另外还有 GetBroadcasts、LocalState 以及 MergeRemoteState 等回调接口可供上层应用实现对接。

其次，当集群成员状态发生变化时，如新节点加入集群、节点主动离开集群或者是节点的元数据更新，协议库通过提供 EventDelegate 接口以使得上层应用提供自己的处理回调逻辑。

再者，当协议库底层执行某些特殊操作时，如新节点加入集群时出现了节点信息冲突，或者是节点间进行全量状态同步时执行合并操作，其也提供了对应的接口以让上层应用知悉此事件的发生，并针对性做出响应。

最后，当协议库在运行协议时，节点收到某些类型的消息，比如节点发送 ack 或者 收到 ack 消息时，其同样会通知上层应用，上层应用可基于此估计网络的 RTT（round-trip-time）值，另外节点收到 Alive 消息也是类似。
